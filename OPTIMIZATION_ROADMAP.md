# Mini Desk Tool 开发计划

本文档用于规划 Mini Desk Tool 后续优化。README 面向用户介绍功能与重大版本演进；本文档面向开发，记录已完成能力、当前收尾项、未来版本计划和发布检查。

## 产品方向

Mini Desk Tool 的定位是桌面边缘的轻量 Launchpad。核心目标不是堆叠功能，而是让用户更快完成三件事：

- 快速打开常用网站、文件、文件夹、应用和快捷方式。
- 低成本整理图标、分组、配置和常用动作。
- 稳定贴合桌面边缘，随用随取，不干扰当前工作。

## 开发原则

- 高频优先：搜索、最近打开、拖拽、分组、右键动作和批量管理优先。
- 低打扰优先：吸附、隐藏、唤出、边缘收起必须稳定，不应遮挡桌面操作。
- 可恢复优先：涉及数据变更的功能必须配套导入、导出、备份、恢复点或确认机制。
- 小步发布：每个版本聚焦一个明确主题，避免一次性混入过多行为变化。
- 可验证优先：每个阶段完成后记录构建、Rust 测试、Playwright 测试或人工验证结果。
- 版本号规则：`v1.2` 作为当前大版本号，常规功能拓展、体验优化和稳定性修复只递增最后一位；只有产品形态或兼容性发生重大变化时，才进入新的大版本段。

## 当前收尾项

当前阶段聚焦 v1.2.8 边缘收起模式的稳定性打磨，不再新增大功能。

- 验证顶部/底部边缘收起是否仍会遮挡底层应用操作。
- 验证多显示器和不同缩放比例下的收起位置、展开方向和动画流畅度。
- 统一窗口行为实验功能区域的文案、颜色和层级。
- 记录抽屉动画、主界面圆角、毛玻璃透明度在不同主题下的视觉表现。
- 如继续出现遮挡问题，再单独评估“窗口收缩热区”或“点击穿透”方案，避免仓促合入。

## v1.2.9：可访问性与低动效

### 目标

让键盘用户、低动效偏好用户和高对比需求用户也能稳定使用面板。

### 功能方案

- 图标网格支持方向键移动焦点。
- `Enter` 打开当前聚焦图标。
- `Delete` 删除当前聚焦图标，删除前必须确认。
- 搜索框、设置导航、右键菜单和批量工具条提供清晰焦点样式。
- 增加“减少动效”选项，降低页面过渡、图标抖动和边缘收起动画。
- 增加高对比主题或高对比开关。

### 验收标准

- 主要功能无需鼠标也能完成基础操作。
- 焦点在深色、浅色、透明背景下都清晰可见。
- 减少动效不会影响功能状态提示。
- 删除、批量删除等高风险操作必须二次确认或可恢复。

## v1.2.10：发布与更新体验

### 目标

降低发布遗漏风险，让用户更容易了解版本变化和反馈问题。

### 功能方案

- 设置页显示当前应用版本。
- 增加“查看更新”入口，跳转 GitHub Releases 或后续更新服务。
- 增加版本更新说明入口。
- 增加本地发布检查脚本，检查版本号一致性。
- 发布前检查 README、开发计划和安装包文件名。
- 增加问题反馈摘要生成入口，自动包含应用版本、系统信息和关键设置，不包含用户数据。

### 验收标准

- `package.json`、`package-lock.json`、`src-tauri/Cargo.toml`、`src-tauri/tauri.conf.json` 版本号一致。
- 更新检查失败时有明确提示。
- 反馈摘要不包含用户图标、路径、备份内容或配置数据。
- 发布包、Git tag 和 GitHub Release 名称保持一致。

## v1.2.11：分组整理增强

### 目标

降低大量图标的整理成本，减少手动拖拽和重复维护。

### 功能方案

- 分组内排序：名称、类型、最近打开时间、添加时间。
- 一键整理分组空位。
- 重复项检测，按 URL、路径、标题提示可能重复项。
- 自动分组建议：开发、设计、视频、办公、文档等。
- 分组标记或轻量颜色标签。
- 空分组策略：始终保留、手动清理、自动隐藏。

### 验收标准

- 排序和整理前提供确认或恢复方案。
- 重复项检测只给建议，不直接删除数据。
- 自动分组只生成建议，不直接覆盖用户现有分组。
- 空分组策略不允许界面变成无分组状态。

## v1.2.12：图标资源管理

### 目标

提升图标加载质量，减少网络图标失败、模糊或重复解析带来的维护成本。

### 功能方案

- 单个图标刷新。
- 批量刷新选中图标。
- 展示图标来源、更新时间和失败原因。
- 清理无效图标缓存。
- 为本地文件、文件夹和应用生成更稳定的图标快照。
- 图标加载失败时显示明确占位和原因。

### 验收标准

- 图标刷新失败不影响项目打开。
- 自定义图标优先级高于自动图标。
- 清理缓存不删除用户自定义图标。
- 批量刷新过程不阻塞主界面。

## v1.2.13：数据恢复增强

### 目标

进一步提升备份可理解性和恢复可控性。

### 功能方案

- 备份列表展示创建时间、项目数、配置数和文件大小。
- 恢复前提供预览：新增、删除、修改多少项目。
- 支持给备份添加备注。
- 支持手动锁定重要备份，避免被自动清理。
- 支持最近高风险操作历史，例如批量删除、导入覆盖、配置删除。
- 支持撤销最近一次高风险操作。

### 验收标准

- 恢复前能看清影响范围。
- 锁定备份不会被保留数量策略删除。
- 撤销失败时不影响当前可用数据。
- 操作历史不记录敏感文件内容，只记录摘要。

## v1.2.14：性能与稳定性

### 目标

让大量图标、多显示器、不同缩放比例和低性能设备下的体验更稳定。

### 功能方案

- 大量图标下优化渲染和拖拽性能。
- 长列表设置页继续优化滚动和定位体验。
- 多显示器与缩放坐标回归测试补充。
- 降低边缘收起、毛玻璃和阴影效果在低性能设备上的成本。
- 增加低性能模式：减少模糊、阴影和动画。

### 验收标准

- 大量图标下拖拽不明显卡顿。
- 低性能模式不改变核心功能行为。
- 多屏缩放下窗口不会跳到错误位置。
- 回归测试覆盖吸附、边缘收起、自动隐藏和全局快捷键关键路径。

## 通用开发流程

1. 确认范围：列出本版本必须做、暂不做和可能影响旧行为的内容。
2. 写交互草案：明确入口、状态、空状态、错误状态和恢复路径。
3. 拆实现任务：前端状态、渲染、Rust 命令、样式和测试分别拆开。
4. 完成最小可用版本：优先跑通主流程，再补边界。
5. 处理边界场景：空数据、大量数据、失败、取消、重复操作和多显示器。
6. 验证：至少执行 `cargo check`、相关 Rust 测试、`npm run build:renderer`，必要时补 Playwright。
7. 更新文档：README 只同步重大功能和版本演进；开发计划同步路线状态。
8. 构建安装包：发布前执行 `npm run build`。
9. 按用户指令决定是否提交、推送或发布 GitHub Release。

- `OPTIMIZATION_ROADMAP.md` 已更新。
- `cargo check` 通过。
- 相关 `cargo test` 通过。
- `npm run build:renderer` 通过。
- 需要发布时，`npm run build` 成功生成安装包。
- 安装包文件名、Git tag 和 Release 名称一致。
- 已记录未覆盖测试或人工验证风险。
